# 需求文档：A 股申万行业成交与买卖分析

> 版本：**v3.5**（管理页 **区间补数** + v3.4 能力）  
> 状态：Phase 1 ✅ · Phase 2/3/3b ✅（持续迭代）  
> 必盈接入：[BIYING_API.md](./BIYING_API.md)  
> 实现方案：[IMPLEMENTATION_PLAN.md](./IMPLEMENTATION_PLAN.md)

---

## 1. 项目目标

基于 **必盈 API**，构建 A 股 **成交额 + 主买/主卖** 与 **ETF 成交额** 的多层级分析能力，并通过 **Web 看板** 展示 **近 5 个交易日** 的趋势与对比。

| 层级 | 指标 | 看板入口 |
|------|------|----------|
| 全 A | 成交额、主买、主卖 | 页面 1 |
| 申万板块（**二级**，约 131 行） | 成交额、占比、主买、主卖 | 页面 2、3 |
| 个股 | 成交额、主买、主卖 | 页面 4（从页面 3 下钻） |
| ETF | 成交额、占 A 股比 | 页面 5、6 |
| 数据管理 | 定时任务、补数、导出、日历 | **页面 7（管理页）** |

**统一时间窗口**：看板页面 1–6 均为 **近 5 个交易日**（非自然日）。

## 2. 已完成能力（Phase 1）

### 2.1 数据源

| 角色 | 平台 | 认证 |
|------|------|------|
| 主数据 | [必盈 API](https://www.biyingapi.com/doc_hs) | `BIYING_LICENCE` |
| 部署 | 腾讯云 Ubuntu + Python 3.10 | `pip install -r requirements.txt` |

### 2.2 采集脚本

`scripts/fetch_by_daily.py` + `scripts/by_common.py`

| 步骤 | 接口 | 输出 |
|------|------|------|
| 全 A 列表 | `hslt/list` | 对照 / 未归类 |
| 申万映射 | `hszg/list` + `hszg/gg`（**type2=1 二级**） | `sector_stock_mapping.csv` |
| 个股成交 | `hsrl/ssjy_more` | `turnover` |
| 个股买卖 | `hsstock/history/transaction` | `active_buy` / `active_sell` |
| 板块 / 全 A 汇总 | 本地聚合 | `sector_*_daily.csv`、`market_summary_daily.csv` |
| ETF 成交 | `fd/list/etf` + `fd/real/time` | `etf_turnover_latest.csv`（**无买卖**） |

### 2.3 板块层级（申万）

| 层级 | 必盈 `type2` | 板块数 | 默认 |
|------|-------------|--------|------|
| 一级 | `0` | 31 | 否（已弃用为看板默认） |
| **二级** | `1` | **131** | **是** |
| 三级 | — | 0 | 必盈无 L3 |

- 默认采集与看板展示：**申万二级**（`scripts/sector_config.py` → `DEFAULT_SECTOR_LEVEL=l2`）。
- 全 A 覆盖率：约 **99.81%**（5198/5208）；约 10 只新股/未入库成份无 L2 归属，与 L1 未归类集合相同。
- 已有 L1 历史库数据：运行 `build_sector_mapping.py --level l2` + `migrate_sectors_to_l2.py` 可**不重打 API** 迁移板块归属并重聚合。
- **sector / stock 一致**：落库或重聚合后自动 **删除** 当日 `sector_daily` 中无对应 `stock_daily` 成份的僵尸行（如 L1 残留 `sw_ysjs`）；亦可运行 `rebuild_sector_aggregates.py` 一键修复。

### 2.4 数据一致性（sector ↔ stock）

| 规则 | 说明 |
|------|------|
| 聚合来源 | `sector_daily` 由 `stock_daily` 按 `sector_code` 汇总 |
| 下钻条件 | 个股页按 `sector_code` 精确匹配；**两边 code 必须一致** |
| L1→L2 | L2 无「有色金属」等 L1 code；迁移后须清理 sector 僵尸行 |
| 修复方式 | 管理页 **成功** 重拉该日；或 `rebuild_sector_aggregates.py` |

### 2.5 当前局限

| 局限 | 影响 |
|------|------|
| 历史不足 5 日 | 看板展示实际 N 日并提示 |
| 必盈无 ETF 买卖 | ETF 页仅成交额 |
| 必盈无申万三级 | L3 需 BigQuant/AKShare 等（Phase 4） |

## 3. 看板通用规则

### 3.1 数值单位

所有金额类展示（图表轴、表格、Tooltip）按数值大小 **自动选用单位**：

| 绝对值范围 | 单位 | 示例 |
|------------|------|------|
| ≥ 1 亿 | **亿** | `123.45 亿` |
| ≥ 1 千万且 < 1 亿 | **千万** | `876.5 千万` |
| ≥ 1 万且 < 1 千万 | **万** | `456.7 万` |
| < 1 万 | **元** | `9,876 元` |

- 存储与 API 返回一律 **元（float）**；格式化仅在前端展示层。
- 占比：**百分比**，保留 2 位小数，如 `12.34%`。
- 同一页面内图表 Y 轴、表格、Tooltip 共用一套格式化函数。

### 3.2 交易日口径

- 「近 5 日」= 数据库中最近 **5 个交易日**（按 `trade_date` 降序取 5 条）。
- 某日无数据则跳过，不插零；不足 5 日时展示已有全部日期，页脚标注「当前 N 个交易日」。
- 必盈买卖数据 **21:30** 更新；默认定时 **21:35**，保证买卖与成交可对齐。

### 3.3 占比口径

| 场景 | 公式 |
|------|------|
| 板块占 A 股（页面 2） | `sector.turnover / market.turnover` |
| ETF 占 A 股（页面 5） | `etf.turnover / market.turnover` |

分母均为 **当日全 A 成交额**（`market_daily.turnover`），非 ETF 市场合计。

### 3.4 导航结构

```text
页面 1  全 A 概览
页面 2  板块表格 ──► 页面 3  板块图表 ──► 页面 4  板块个股
页面 5  ETF 表格
页面 6  ETF 图表
页面 7  管理（定时 / 补数 / 导出 / 数据日历）
```

**顶栏导航（建议）**

```text
[概览] [板块表格] [板块图表] [ETF 表格] [ETF 图表] [管理]
```

- 页面 3 每板块 **「查看个股」** → 页面 4 `/sectors/{sector_code}/stocks`。
- 页面 4 面包屑：`概览 > 板块图表 > {板块名} > 个股`。

### 4.7 页面 7：数据管理

**路由**：`/admin`  
**权限**：仅管理员可访问（Basic Auth 或独立 admin token；与普通看板只读分离）。

管理页集中提供 **采集调度、手动补数、数据导出、数据日历** 四类能力，并统一展示 **任务状态与报错信息**。

#### 4.7.1 定时更新配置

| 项 | 说明 |
|----|------|
| 功能 | 配置 **每天自动更新** 当日（最新可用 `trade_date`）全量数据的执行时刻 |
| 默认时间 | **21:35**（Asia/Shanghai） |
| 执行日类型 | **交易日**（默认）或 **自然日**；管理页可切换 |
| 配置项 | 时、分（24 小时制）；是否启用；执行日类型（`trading_day` / `calendar_day`） |
| 交易日模式 | 仅在 **A 股交易日** 触发；周末与法定休市日跳过 |
| 自然日模式 | **每天**到点触发（含周末/节假日）；适用于测试或强制补跑 |
| 更新范围 | 与 `fetch_by_daily.py` 一致：行业映射缓存、个股成交/买卖、板块汇总、全 A 汇总、ETF 成交，并 UPSERT `history.db` |
| 状态展示 | 最近一次定时任务：开始/结束时间、目标 `trade_date`、状态（成功/失败/运行中）、耗时 |
| 报错展示 | 失败时展示 **错误摘要** + **日志尾部**（可展开完整 log）；支持复制 |

**说明**

- 界面「当日」指任务运行时 API 能提供的 **目标交易日期**（`trade_date`），写入 DB 以该日期为键。
- **21:35** 为产品缺省，与必盈买卖 21:30 更新衔接；管理员可在管理页调整。

#### 4.7.2 手动指定日期 / 区间补数

| 项 | 说明 |
|----|------|
| 功能 | 填写 **开始日期** 与 **结束日期**（均为 `YYYY-MM-DD`，**两项必填**），手动触发该闭区间内 **全部 A 股交易日** 的数据采集并写入历史库 |
| 单日 | **开始 = 结束** 时视为补 **1 个交易日**（与旧版「选一天」等价） |
| 区间展开 | 系统用 SSE 交易日历计算 `[开始, 结束]` 内交易日列表；自然日中的周末/休市 **自动跳过** |
| 预览 | 提交前展示「共 **N** 个交易日」及首尾日期；**N = 0** 时禁止提交 |
| 覆盖策略 | **不跳过** 库内已有完整数据的日期；区间内每个交易日 **全量重拉并 UPSERT 覆盖** |
| 校验 | 见下表；管理页对开始/结束日分别显示「✓ 交易日 / ✗ 休市」（仅提示，不阻止区间边界为休市日） |
| 场景 | 新库初始化、长假后批量补漏、定时任务连续失败后一次性回填 |
| 交互 | 开始日期 + 结束日期 +「开始更新」；运行中可 **取消** 任务 |
| 状态展示 | 任务队列：ID、日期范围、触发方式、状态（含 `cancelled`）、进度、耗时 |
| 报错展示 | 同 4.7.1；失败任务可 **一键重试**（仍校验区间） |
| 并发 | 同一时刻 **仅 1 个采集任务**；第二个返回 409 |

**输入校验**

| 规则 | 行为 |
|------|------|
| 开始、结束均必填 | 缺任一项 → 400 |
| 开始 ≤ 结束 | 否则 → 400「结束日期不能早于开始日期」 |
| 结束 ≤ 今天 | 不允许补未来日期 → 400 |
| 区间内交易日数 | **1 ~ 30**（上限可配置）；超出 → 400 |
| 区间内无交易日 | → 400「所选区间无 A 股交易日」 |
| 单日且非交易日 | **拒绝**（休市无行情/资金流，补数无意义） |
| 区间边界为休市日 | **允许**（自然日范围可跨周末；实际只补中间交易日） |

**数据拉取口径（区间内每个交易日）**

| 数据 | 接口策略 |
|------|----------|
| 买卖 | 按股 `history/transaction?st=YYYYMMDD&et=YYYYMMDD`（区间一次拉取后按日拆分）；实现见方案文档 |
| 成交额 | 按股日 K `hsstock/history/{code}/d/n?st=&et=` |
| 当日（区间含今天） | 今天可走 `fetch_by_daily` 实时路径；历史日用 K 线 + transaction |
| ETF | 仍仅 **今天** 可 API 补；历史 ETF 不能通过区间补数批量获取，任务日志 **warning**，该日 `etf_daily` 可能仍为空 |
| 落库 | 按日 UPSERT 四表；sector 写入后 **prune** 僵尸行（同 v3.4） |

**任务进度（区间）**

- `progress` 示例：`stocks 1200/5208` 或 `dates 3/8`
- 任务记录 **日期** 列展示：`2026-06-01 ~ 2026-06-12`（单日则 `2026-06-12`）

**与 `backfill_history.py` 区别**

| | 区间补数（4.7.2） | `backfill_history --days N` |
|--|-------------------|----------------------------|
| 日期范围 | 用户指定起止 | 从今天往前 N 个交易日 |
| 入口 | 管理页 | CLI |
| 跳过已有 | **否**，全量覆盖 | 否（UPSERT 覆盖） |

#### 4.7.3 按日下载全量数据

| 项 | 说明 |
|----|------|
| 功能 | 选择某一已有数据的 `trade_date`，下载 **该日全部数据** 为文件包 |
| 格式 | **ZIP**，内含 UTF-8 **CSV** + `README.txt` |
| 文件清单 | `market_daily.csv`（1 行）、`sector_daily.csv`、`stock_daily.csv`、`etf_daily.csv`、`meta.json`（行数、生成时间、trade_date） |
| 交互 | 日期下拉（仅列出已有数据的日期）+「下载」按钮 |
| 空数据 | 该日无记录时按钮禁用并提示 |

#### 4.7.4 已有数据日期日历

| 项 | 说明 |
|----|------|
| 功能 | 展示系统中 **已落库** 的日期列表及完整度，便于掌握数据情况 |
| 展示形式 | **日历视图** + **列表视图**（可切换） |
| 每个日期显示 | `trade_date`；是否有 market / sector / stock / etf；各行数；最近更新时间 |
| 完整度标识 | ✅ 完整（四表均有且 stock_count > 0）／⚠ 部分（缺 ETF 或买卖）／❌ 失败任务残留 |
| 交互 | 点击某日 → 快捷 **下载** 或 **重新采集**（跳转 4.7.2 并预填开始/结束为同一日） |

#### stock_daily（~5200 行 / 日）

| 字段 | 说明 |

## 10. 验收标准（Phase 3）

1. **页面 1**：三张柱图均为近 5 日，字段与 `market_daily` 一致，单位正确。
2. **页面 2**：5 个日组 × 2 列，占比排序默认降序，点击跳转页面 3 有效。
3. **页面 3**：每板块 3 图可折叠，「查看个股」进入页面 4。
4. **页面 4**：仅展示该板块成份股，每股 3 图可折叠。
5. **页面 5**：ETF 表格 5 日组成交 + 占 A 股比，排序与搜索可用，1480 行不卡顿。
6. **页面 6**：每 ETF 1 图可折叠，懒加载；与页面 5 排序/定位一致。
7. 历史不足 5 日时，各页展示实际天数并提示。

### 验收标准（Phase 3b 管理页）

1. 默认定时 **21:35**、执行日类型 **交易日**，修改后持久化并按新规则触发。
2. 手动选择日期触发采集，任务列表实时展示状态；失败可见错误信息与日志。
3. 已有数据的日期可下载 ZIP，内含 market / sector / stock / etf 四表 CSV。
4. 日历/列表正确展示已落库日期与各表行数；部分数据日期标 ⚠。
5. 并发提交两个手动任务时，第二个返回 409；非交易日提交返回 400。
6. 运行中任务可取消，状态变为 `cancelled`，子进程终止。

### 验收标准（Phase 3c 区间补数）

1. 管理页 **开始、结束日期均必填**；相等时补单日；不等时补闭区间内全部交易日。
2. 结束早于开始、结束晚于今天、区间内 0 个交易日、超过 30 个交易日 → 400 且前端提示明确。
3. 提交前展示「共 N 个交易日」预览；**不跳过** 库内已有完整数据的日期，成功后数据被覆盖。
4. 任务列表日期列展示区间；进度可观察（股票数或日序）；长任务可取消。
5. 区间仅含历史日时 A 股买卖+成交落库；含今天时今天走实时路径；历史 ETF 仍缺则标 ⚠。
